# 数据库统一安全管控平台客户演示手册

本文档供公司同事进行客户演示时使用，包含推荐路线、演示账号、操作步骤、SQL 示例、预期结果和客户问答口径。

## 1. 演示目标

这套系统要向客户证明一件事：

> 数据库访问不再是“谁拿到账号谁就能直接查、直接改”，而是统一经过身份、权限、SQL 审核、审批、脱敏和审计。

推荐演示时长：

- 15 分钟：只讲主线，适合客户领导或初次沟通。
- 30 分钟：完整演示，适合需求评审或产品交流。
- 45 分钟：加上规则配置、敏感字段维护、权限配置和问答。

演示主线：

1. 普通用户查询敏感数据，只能看到脱敏结果。
2. DBA 用户有明文授权，但敏感访问仍会留审计。
3. SQL 客户端提供资源树、表结构、SQL 执行和历史回填的轻量体验。
4. 选错库、写错方言、越权查表、高危 SQL 会被阻断。
5. 生产 DML 变更不能直接执行，必须审批。
6. 审批通过后执行前还会二次校验 SQL 和执行窗口。
7. 所有关键动作都能在审计日志里筛选、追溯和导出。
8. 用户权限、规则策略、敏感字段目录都可以配置。

## 7. 开场白

可以照着说：

> 今天我们演示的是数据库统一安全管控平台。它不是替代数据库，而是在用户访问数据库之前加一道统一入口。用户还是输入 SQL，但平台会先判断这个人是谁、能不能访问这个库、能不能操作这张表、SQL 风险高不高、是否要审批、结果要不要脱敏。最后所有动作都会进入审计。

再补一句业务痛点：

> 以前很多客户的问题是数据库账号分散、权限边界不清、敏感数据容易明文暴露、生产数据修改缺少审批和追溯。这个平台就是把这些风险统一收口。

## 8. 流程一：敏感查询与动态脱敏

### 要证明什么

普通研发可以查询授权范围内的数据，但姓名、身份证号、手机号、地址等敏感字段会自动脱敏。

### 操作步骤

1. 登录 `zhangming` / `123456`。
2. 左侧进入 `SQL 安全` > `SQL审核工作台`。
3. 目标库选择 `客户中心-MySQL生产库`。
4. SQL 输入框粘贴：

```sql
select name, id_card, mobile, address, account_balance from customer_info where city = '北京' limit 20;
```

5. 点击右上角 `执行审核`。
6. 在 `审核结论` 页签看：
   - SQL 类型是 `SELECT`。
   - 目标库是 `客户中心-MySQL生产库`。
   - 目标表是 `customer_info`。
   - 处置结果是放行。
   - 带有 `动态脱敏` 标签。
7. 切到 `规则命中` 页签，指出 `R-COM-007 敏感字段动态脱敏`。
8. 切到 `查询结果` 页签，观察：
   - 姓名类似 `张*`。
   - 身份证号中间是星号。
   - 手机号类似 `138****5678`。
   - 地址后半部分隐藏。
   - 账户余额正常展示，不作为金额脱敏示例。
9. 进入 `SQL 安全` > `审计日志`。
10. 通过 `用户 = 开发用户 张明` 或 `动作 = SQL执行` 筛选，展示本次查询记录。

### 讲解话术

> 张明是普通研发。他可以查授权范围内的客户表，但平台识别到 SQL 里包含姓名、证件号、手机号、地址，所以查询可以执行，结果中的这些字段必须脱敏。账户余额这类金额字段在当前演示口径里不做脱敏规则展示。

### 演示提示

- `select` 就是查询。
- `customer_info` 可以理解成“客户信息表”。
- `where city = '北京'` 表示只查北京客户。
- `limit 20` 表示最多返回 20 行。

### 如果结果不符合预期

- 如果没有脱敏，确认当前用户是不是 `zhangming`。
- 如果目标库不是 MySQL，重新选择 `客户中心-MySQL生产库`。
- 如果 SQL 没复制完整，重新复制上面的整段 SQL。

## 9. 流程二：DBA 明文授权对照

### 要证明什么

同一条 SQL，不同用户权限不同。DBA 可以看明文，但敏感访问仍然留审计。

### 操作步骤

1. 点击右上角用户名称，切换到 `DBA 管理员 王强`。
2. 进入 `SQL 安全` > `SQL审核工作台`。
3. 目标库选择 `客户中心-MySQL生产库`。
4. 粘贴同一条 SQL：

```sql
select name, id_card, mobile, address, account_balance from customer_info where city = '北京' limit 20;
```

5. 将工作台上方脱敏开关切到 `明文`。
6. 点击 `执行审核`。
7. 查看 `查询结果`：
   - 姓名、手机号、证件号、地址等可以看到明文。
8. 进入 `审计日志`，查看本次敏感访问记录。

### 讲解话术

> 明文不是所有人都能看，而是按用户授权控制。王强是 DBA，有明文授权，所以同一条 SQL 可以返回明文。但系统仍然记录敏感访问审计，后续可以追溯是谁、什么时候、在哪个库、看了什么。

### 客户容易问

客户问：“DBA 能看明文，会不会还是有风险？”

回答：

> 平台不是只靠脱敏，而是把明文权限、敏感访问审计、用户身份、SQL 内容和访问时间放在一起管。DBA 能看是授权结果，不是绕过管控。

## 10. 流程三：目标库选择会影响结果和审核

### 要证明什么

目标库不是装饰项。相同 SQL 在不同目标库下，结果、规则和风险可能不同。

### 操作步骤 A：同一客户表，不同目标库返回不同业务数据

1. 当前使用 `DBA 管理员 王强`。
2. 进入 `SQL 安全` > `SQL审核工作台`。
3. 目标库选择 `客户中心-MySQL生产库`。
4. 粘贴：

```sql
select customer_id, name, mobile, city from customer_info where city = '北京' limit 20;
```

5. 点击 `执行审核`。
6. 记住查询结果里的客户号，通常是 `C1001`、`C1004`、`C1006`。
7. 把目标库切换为 `实时风控-GaussDB生产库`。
8. SQL 不变，再点 `执行审核`。
9. 观察查询结果变化，客户号通常变为 `RC3001`。

讲解话术：

> 两个库里都可能有 `customer_info` 这张表，但业务含义不同。客户中心库是客户主数据，风控库是风险客户快照。平台会按所选目标库取对应数据，也会按所选目标库做权限和规则判断。

### 操作步骤 B：同一 SQL 切到 Oracle 后被阻断

1. 目标库切换为 `核心账务-Oracle生产库`。
2. SQL 保持：

```sql
select customer_id, name, mobile, city from customer_info where city = '北京' limit 20;
```

3. 点击 `执行审核`。
4. 预期结果：
   - 审核结论变为阻断。
   - 规则命中 Oracle 方言校验。

讲解话术：

> 这条 SQL 里的 `limit 20` 是 MySQL 常见写法。Oracle 不用这种分页语法，所以切到 Oracle 目标库后，平台会发现方言不匹配并阻断，避免用户选错库后继续执行。

### 演示提示

- MySQL、Oracle、GaussDB、DB2 是不同数据库类型。
- 不同数据库语法不完全一样，像普通话和方言的区别。
- 目标库决定“按哪种数据库规则检查 SQL”。

## 11. 流程四：高危和越权阻断

### 要证明什么

平台不仅会脱敏，也会阻断危险 SQL。这里建议连续演示四类阻断：条件高危、库表不匹配、操作类型越权、数据库方言不匹配。

### 场景 A：恒真条件阻断

1. 切换到 `风控用户 刘伟`。
2. 进入 `SQL 安全` > `SQL审核工作台`。
3. 目标库选择 `客户中心-MySQL生产库`。
4. 粘贴：

```sql
select * from customer_info where 1=1;
```

5. 点击 `执行审核`。
6. 预期结果：
   - 处置结果为阻断。
   - 风险等级为严重风险。
   - 命中 `WHERE 恒真条件拦截`。

讲解话术：

> `where 1=1` 对数据库来说永远成立，容易变成大范围查询或绕过条件。平台识别到这种高危特征后直接阻断。

### 场景 B：目标库和表不匹配阻断

1. 仍使用刘伟。
2. 目标库选择 `客户中心-MySQL生产库`。
3. 粘贴：

```sql
select employee_name, salary_amount from employee_salary where department_id = 12 limit 20;
```

4. 点击 `执行审核`。
5. 预期结果：
   - 直接阻断。
   - 命中 `R-COM-004 授权表范围校验`。

讲解话术：

> `employee_salary` 是人事薪酬表，不属于客户中心 MySQL 库的授权表范围。即使用户会写 SQL，也不能通过选错库去查别的业务域数据。

### 场景 C：只读用户提交 DML 被阻断

1. 仍使用 `风控用户 刘伟`。
2. 目标库选择 `实时风控-GaussDB生产库`。
3. 粘贴：

```sql
update risk_event set event_status = 'CLOSED' where event_id = 'RISK202606010001';
```

4. 点击 `执行审核`。
5. 预期结果：
   - 直接阻断，不生成审批单。
   - SQL 类型识别为 `UPDATE`。
   - 目标表识别为 `risk_event`。
   - 命中 `R-COM-003 操作类型授权`。
   - 提示刘伟所在角色不具备 `UPDATE` 操作权限。

讲解话术：

> 刘伟可以看风控数据，但他的角色是只读分析师，没有生产变更权限。所以这条 SQL 不会进入审批，而是在权限校验阶段就被拒绝。平台先判断“这个人能不能做这类操作”，再判断后续是否需要审批。

### 场景 D：Oracle 方言不匹配阻断

1. 切换到 `DBA 管理员 王强`。
2. 目标库选择 `核心账务-Oracle生产库`。
3. 粘贴：

```sql
select * from customer_info limit 10;
```

4. 点击 `执行审核`。
5. 预期结果：
   - 直接阻断。
   - 命中 Oracle 方言校验。

讲解话术：

> 平台会结合目标库类型做语法校验，Oracle 目标库不接受 MySQL 的 `LIMIT` 写法，所以不会让这条 SQL 继续执行。

### 本流程结束时可以总结

> 这几类阻断证明平台不是简单做 SQL 文本记录，而是在执行前做了身份、数据源、表范围、操作类型、语法方言和风险规则的组合校验。只有全部通过，SQL 才会进入放行、脱敏或审批流程。

## 12. 流程五：DML 审批闭环

### 要证明什么

生产数据变更不能直接执行。INSERT、UPDATE、DELETE 等 DML 命中生产库 DML 审批规则后，会生成审批单；审批通过后由申请人执行；执行前还会校验 SQL 指纹和执行窗口。

### 推荐主线：UPDATE 审批闭环

客户现场建议用 UPDATE 做完整闭环，因为 UPDATE 更容易解释“修改生产数据”这件事。

### 为什么推荐用王强发起

李娜的执行窗口是 `20:00-23:00`。如果客户现场不是这个时间，李娜提交变更会被执行窗口规则阻断。为了保证现场稳定，推荐：

- 王强发起变更。
- 李娜审批。
- 王强执行。

这样任何时间都能演示完整闭环。

### 操作步骤

1. 切换到 `DBA 管理员 王强`。
2. 进入 `SQL 安全` > `SQL审核工作台`。
3. 目标库选择 `实时风控-GaussDB生产库`。
4. 粘贴：

```sql
update risk_event set event_status = 'CLOSED' where risk_level = 'LOW' and event_date < '2026-06-01';
```

5. 点击 `执行审核`。
6. 预期结果：
   - SQL 类型是 `UPDATE`。
   - 目标表是 `risk_event`。
   - 命中 `R-COM-006 生产库 DML 审批`。
   - 处置结果是进入审批。
   - 弹出 `生成 DML 审批单` 表单。
7. 审批单填写：
   - 审批人：`运维用户 李娜`
   - 变更原因类别：`风险事件归档`
   - 变更原因：`批量关闭低风险历史事件，避免模型看板继续统计已过观察期的事件。`
   - 预计影响行数：`128`
   - 回滚方案：`执行前备份 risk_event 命中记录；如需回滚，按备份主键恢复 event_status。`
8. 点击 `生成审批单`。
9. 切换到 `运维用户 李娜`。
10. 进入 `SQL 安全` > `DML审批` > `审批列表`。
11. 找到刚生成的审批单，点击 `通过`。
12. 切换回 `DBA 管理员 王强`。
13. 进入 `SQL 安全` > `DML审批` > `审批列表`。
14. 找到已通过的审批单，点击 `执行`。
15. 进入 `DML审批` > `执行列表`，查看执行记录。
16. 进入 `审计日志`，按 `模块 = DML审批` 筛选。

### 每一步怎么讲

生成审批单时：

> 生产 UPDATE 不允许直接执行，平台自动把它转成审批单。审批材料里必须写清楚变更原因、预计影响和回滚方案。

李娜审批时：

> 审批人看到的不只是“同意/拒绝”，还包括 SQL、目标库、目标表、风险等级、影响行数和回滚方案。

王强执行时：

> 审批通过不等于永久放权，只对这张审批单里的 SQL 生效。执行前平台会校验 SQL 指纹，防止审批后被篡改。

看执行列表时：

> 执行列表保留了执行人、执行时间、影响行数、回滚方案和 SQL 指纹，后续可以做变更复盘。

看审计日志时：

> 这条变更从提交审批、审批通过到最终执行，都形成了完整审计链路。

### 补充场景：INSERT 也会进入审批

如果客户问“新增数据是不是也管”，按这个补充演示。这个场景只需要展示审核结论和审批单生成口，不一定现场走完整审批执行。

1. 切换到 `DBA 管理员 王强`。
2. 进入 `SQL 安全` > `SQL审核工作台`。
3. 目标库选择 `实时风控-GaussDB生产库`。
4. 粘贴：

```sql
insert into risk_event (event_id, customer_id, risk_level, event_status, scene, score, event_date, handler) values ('RISK202606180088', 'RC3001', 'MEDIUM', 'OPEN', '人工复核新增', 72, '2026-06-18', '风控平台');
```

5. 点击 `执行审核`。
6. 预期结果：
   - SQL 类型识别为 `INSERT`。
   - 目标库是 `实时风控-GaussDB生产库`。
   - 目标表是 `risk_event`。
   - 命中 `R-COM-006 生产库 DML 审批`。
   - 处置结果是进入审批。
   - 弹出 `生成 DML 审批单` 表单。
7. 可以填写：
   - 审批人：`运维用户 李娜`
   - 变更原因类别：`生产数据修正`
   - 变更原因：`风控人员人工复核后，需要补录一条风险事件用于后续跟踪。`
   - 预计影响行数：`1`
   - 回滚方案：`delete from risk_event where event_id = 'RISK202606180088';`
8. 如果时间紧，停在审批单表单即可；如果客户要求看完整闭环，再点击 `生成审批单`，切到李娜审批，通过后切回王强执行。

讲解话术：

> DML 不只包括 UPDATE，INSERT 和 DELETE 也属于生产数据变更。平台会先识别 SQL 类型，再结合目标库、目标表、用户权限和规则策略决定是否进入审批。这里新增一条风控事件，影响行数只有 1 行，但因为它是生产库 DML，仍然必须走审批。

### 如果现场被执行窗口阻断

如果用李娜发起 DML，当前时间不在 `20:00-23:00`，会被阻断。这不是故障，而是规则生效。

可以这样说：

> 这里正好能看到执行窗口控制。生产运维不是随时都能提交或执行变更，必须在授权窗口内操作。

然后切回王强，按上面的稳定路线继续完整闭环。

讲解话术：

> 客户可以先使用平台预置规则，后续按内部制度调整。例如哪些规则阻断、哪些规则提示、哪些规则进入审批，都可以配置。

## 17. 流程十：SQL 客户端轻量体验

### 要证明什么

平台不只是一个 SQL 审批表单，也可以给数据库使用人员提供接近 PL/SQL 工具的轻量操作入口：能浏览数据源资源树、查看表结构、生成查询、执行受控 SQL、回看执行历史。

### 操作步骤

1. 切换到 `DBA 管理员 王强`。
2. 进入 `SQL 安全` > `SQL客户端`。
3. 左侧查看 `数据源资源树`：
   - 资源树来自 `数据库管理` > `连接管理` 当前连接数据。
   - 数据源按 MySQL、Oracle、GaussDB、DB2 展示。
   - 数据源下面按 Schema 展示表。
   - 未授权表会标记 `未授权`，执行时会被平台阻断。
   - 连接或授权表范围变更后，刷新 SQL 客户端即可同步。
   - 新增授权表即使没有专门字段配置，也会生成默认表结构和演示结果，避免现场空白。
4. 在 `客户中心-MySQL生产库` 下选择 `crm_core.customer_info`。
5. 在 `表结构详情` 页签说明这些信息：
   - Schema：`crm_core`
   - 表名：`customer_info`
   - 中文名：客户基础信息表
   - 数据量：`28,643,912`
   - 敏感等级：`高`
   - 字段清单：字段名、类型、是否可为空、字段说明。
   - 敏感字段：`name`、`id_card`、`mobile`、`address` 会显示敏感类型和脱敏规则。
6. 点击 SQL 编辑器右上角 `生成查询`。
7. 系统会生成类似 SQL：

```sql
select customer_id, name, id_card, mobile, address from customer_info where city = '北京' limit 20;
```

8. 点击 `执行`。
9. 查看 `查询结果` 页签：
   - 王强默认可看明文。
   - 如果把编辑器上方开关切到 `脱敏`，再次执行后会看到脱敏结果。
10. 查看 `审核结论` 页签：
    - SQL 类型为 `SELECT`。
    - 目标库为 `客户中心-MySQL生产库`。
    - 目标表为 `customer_info`。
    - 命中敏感字段规则时会展示风险等级和处置。
11. 查看 `执行历史` 页签：
    - 能看到执行时间、用户、数据源、SQL 类型、结果、风险、行数、耗时、SQL。
    - 点击某条历史的 `回填`，SQL 会回到编辑器里，方便复查或再次执行。
12. 进入 `SQL 安全` > `审计日志`，按 `模块 = SQL客户端` 筛选，展示刚才的执行审计。

## 21. 常见翻车处理

### 21.1 登录后菜单不一样

原因：当前用户权限不同。

处理：

1. 点击右上角用户名。
2. 切换到 `DBA 管理员 王强`。
3. 如果仍不对，刷新页面。

### 21.2 查询结果没有脱敏

原因：可能当前是王强，且切到了明文视图。

处理：

1. 切换到 `开发用户 张明`。
2. 确认脱敏开关不是明文。
3. 重新执行 SQL。

### 21.3 DML 没有进入审批，而是被阻断

原因：可能当前用户不在执行窗口内，或 SQL 命中了严重阻断规则。

处理：

1. 使用推荐路线：王强发起、李娜审批、王强执行。
2. 确认 SQL 是本文提供的 `update risk_event ...`。
3. 如果要演示 INSERT，使用本文提供的 `insert into risk_event ...`。
4. 不要用 `delete from customer_info;` 做审批闭环，它会被阻断。
5. 不要用刘伟发起 DML 审批闭环，刘伟是只读用户，会命中操作类型授权阻断。

### 21.4 审批列表看不到通过按钮

原因：当前账号不是审批单指定审批人。

处理：

1. 看审批单的 `处理人`。
2. 切换到对应用户。
3. 推荐闭环里审批人是 `运维用户 李娜`。

### 21.5 审批通过后看不到执行按钮

原因：当前账号不是申请人。

处理：

1. 看审批单的 `申请人`。
2. 切回申请人。
3. 推荐闭环里执行人是 `DBA 管理员 王强`。

### 21.6 目标库切换后结果看起来没变

处理：

1. 用这条更容易看出差异的 SQL：

```sql
select customer_id, name, mobile, city from customer_info where city = '北京' limit 20;
```

2. MySQL 结果通常是 `C1001`、`C1004`、`C1006`。
3. GaussDB 结果通常是 `RC3001`。
4. Oracle 会因为 `LIMIT` 方言不匹配被阻断。

### 21.7 SQL 客户端执行后提示需要审批

原因：SQL 客户端不会绕过平台审批。INSERT、UPDATE、DELETE 等生产 DML 仍然会进入审批策略。

处理：

1. 如果要演示查询能力，使用资源树的 `生成查询`。
2. 如果要演示 DML 闭环，进入 `SQL 安全` > `SQL审核工作台` 生成审批单。
3. 如果客户问为什么客户端不能直接执行 DML，回答：这是平台管控点，不是功能缺失。

### 21.8 页面数据被改乱了

处理：

1. 打开浏览器控制台。
2. 执行：

```js
window.__RKLINK_SQL_SECURITY_DEMO__?.resetAll();
```

3. 刷新页面。

## 22. 客户常见问题回答口径

### Q1：目标库选择会影响后续流程吗？

会。目标库会影响数据源授权、授权表清单、数据库方言、敏感等级、执行窗口和最终策略。比如 Oracle 库里输入 MySQL 的 `LIMIT` 会被方言规则阻断；MySQL 目标库里查 DB2 薪酬表会被授权表范围校验阻断。

### Q2：执行的 SQL 和审批 SQL 不一致怎么办？

审批单会保存原始 SQL 和 SQL 指纹。审批通过后执行前会重新计算当前 SQL 指纹，不一致会阻断并写入 `执行前校验失败` 审计。

### Q3：为什么要填变更原因、预计影响行数、回滚方案？

这是高风险变更的审批材料，便于审批人判断风险，也便于事后审计。当前流程会把这些字段随审批单、执行记录和审计链路一起保存。

### Q4：规则是否可以按客户制度调整？

可以。`SQL规则策略` 页面可维护规则名称、范围、风险等级、处置策略、优先级和状态。后续真实建设时可对接客户制度、审批层级和规则库。

### Q5：自定义规则怎么被平台规则引擎识别？

回答口径：

> 自定义规则不是让管理员写一段中文给系统猜。中文规则名称和说明是给人看的，真正执行时会转成平台规则引擎能识别的结构化条件。

可以这样展开：

1. 平台先把 SQL 解析成结构化信息，例如 SQL 类型、目标库、目标表、字段清单、是否有 `WHERE`、是否命中敏感字段、是否窗口外执行。
2. 自定义规则通过条件配置表达，例如 `SQL 类型 = UPDATE`、`目标环境 = 生产`、`是否包含 WHERE = 否`、`预计影响行数 > 1000`。
3. 规则引擎用这些结构化条件匹配解析结果，命中后输出处置动作：提示、脱敏、审批或阻断。

示例：

```text
规则：UPDATE / DELETE 必须带 WHERE
条件：SQL 类型属于 UPDATE、DELETE，并且未识别到 WHERE
动作：阻断
```

如果用户提交：

```sql
delete from customer_info;
```

平台解析出 `SQL 类型 = DELETE`、`hasWhere = false`，就会命中这条规则并阻断。

演示口径补充：

> 当前演示里，内置规则是真实参与审核的；新增自定义规则主要展示配置模型和审计留痕。正式建设时，自定义规则会落成结构化规则 DSL 或条件编排器，由后端规则引擎统一执行。

### Q6：敏感字段和脱敏规则是否可配置？

可以。`数据资产与脱敏` 的 `敏感字段目录` 可维护字段表达式、敏感类型、等级、脱敏规则、示例和状态。启用状态会参与后续敏感字段识别，变更也会留审计。

### Q7：是否支持审计导出？

支持。审计日志可以按时间、模块、动作、用户、数据源、SQL 类型、处置、风险、命中规则筛选，并导出 CSV。导出动作本身也会写审计。

### Q8：现在看到的是完整生产系统吗？

回答口径：

> 当前是用于客户交流和需求确认的可交互原型，重点展示业务闭环和管控逻辑。真实建设时会接入客户的统一身份认证、数据库连接代理、SQL 解析服务、审批流服务、审计存储和脱敏策略引擎。

### Q9：如何保证用户不能绕过平台直接操作生产库？

回答口径：

> 这件事不能只靠前端页面实现，正式落地时要靠网络、账号和数据库权限一起收口。平台作为统一入口负责身份识别、SQL 解析、权限判断、审批、脱敏和审计；同时生产数据库侧要配合做直连限制，确保用户不能拿个人账号绕开平台。

可以展开成三层：

1. 网络层：生产库只允许平台服务、数据库代理或堡垒机网段访问，普通办公网、研发机、个人电脑不能直接连生产库端口。
2. 账号层：回收个人直连生产库账号，数据库连接凭据由平台统一托管和轮换，用户登录平台但不接触真实数据库密码。
3. 权限层：平台侧按用户、角色、数据源、表、字段、操作类型授权；数据库侧账号也按最小权限配置，必要时通过代理或临时授权执行审批通过的 SQL。

一句话总结：

> 平台管控的是访问入口和执行链路，网络和账号治理负责防止绕路。只有把生产库直连入口收掉，平台的 SQL 审核、审批、脱敏和审计才真正形成闭环。

如果客户追问“现在客户内部还有人知道数据库账号怎么办”，回答：

> 那就需要作为上线前治理事项处理：梳理现有账号、回收或降权个人账号、限制来源 IP、轮换生产密码，再把生产访问逐步迁到平台。否则任何系统都无法阻止一个仍然持有生产账号和网络通路的人绕开平台。

### Q10：是否必须懂 SQL 才能审批？

不一定。审批单里会展示目标库、目标表、风险等级、命中规则、预计影响、变更原因和回滚方案。业务审批人可以先判断变更必要性和影响范围，DBA 或安全人员再判断 SQL 风险。

### Q11：阻断的 SQL 能否管理员特批？

当前演示口径建议：

> 严重风险 SQL 默认不进入普通审批，必须修改 SQL 或走更高等级的线下变更流程。是否允许特批，可以在后续需求阶段由客户制度决定。

### Q12：SQL 客户端是不是替代 PL/SQL？

不是完全替代。当前演示里的 `SQL客户端` 是轻量入口，重点覆盖客户最常问的四件事：数据源资源树、表结构详情、受控 SQL 执行、执行历史回填。

建议回答：

> PL/SQL Developer 这类工具更偏数据库开发和运维工作台，能力包括对象管理、调试、会话管理、执行计划、脚本批量执行等。我们的平台首先解决的是生产访问管控：谁能访问、能访问哪些库表、SQL 是否安全、是否要审批、结果是否脱敏、行为是否审计。轻量 SQL 客户端是为了降低使用门槛，但不会绕过平台规则。

如果客户明确要求完整 PL/SQL 能力，可以这样承接：

> 后续可以继续补对象详情增强、执行计划、SQL 格式化、脚本模板、会话查看、收藏夹、结果导出审批等能力。但生产 DML、敏感字段和高危 SQL 仍然要经过平台审核、审批、脱敏和审计。

### Q13：SQL 客户端的数据源资源树从哪里来？

来自 `数据库管理` > `连接管理` 的连接数据。演示版的 mock 数据已经做了联动：连接新增、编辑、删除，或授权表范围变化后，SQL 客户端刷新资源树会同步变化。

讲给客户时可以说：

> SQL 客户端不是单独维护一套库表清单，而是复用平台登记的数据源和授权范围。这样能保证“连接管理看到什么、权限分配授权什么、客户端就按什么展示和校验”。

### Q14：权限分配改了以后，SQL 客户端会跟着变吗？

会。角色里的数据源范围、SQL 操作类型、明文授权、动态脱敏策略都会影响 SQL 客户端。

最简单的演示方式：

1. 到 `数据安全管理` > `权限分配`，给某个角色补充一个数据源或表范围。
2. 保存后切到该角色成员账号。
3. 进入 `SQL客户端` 刷新资源树。
4. 新授权范围会出现在资源树里，执行 SQL 时也按新权限审核。

### Q15：新增授权表没有专门维护表结构，演示会不会空？

不会。演示版做了兜底 mock：如果新增授权表没有专门配置字段，系统会根据表名和数据库类型生成默认表结构；执行查询时也会生成少量示例结果。

但正式项目建议接入真实元数据采集：

> 正式环境会从数据库字典、元数据扫描或资产目录同步表结构、字段类型、敏感等级和责任人，不依赖手工 mock。

### Q16：SQL 指纹是什么？

SQL 指纹可以理解成这条 SQL 的“唯一摘要”。平台在生成审批单时记录一次，执行前再算一次。

如果审批时是：

```sql
update risk_event set event_status = 'CLOSED' where risk_level = 'LOW';
```

执行前有人改成：

```sql
update risk_event set event_status = 'CLOSED';
```

指纹就会不一致，平台会阻断并写审计。它解决的是“审批通过后偷偷改 SQL”的问题。

### Q17：恒真条件为什么要阻断？

恒真条件就是永远成立的条件，比如 `where 1=1`。它容易造成两类风险：

- 查询条件失效，变成大范围查询。
- UPDATE 或 DELETE 条件失效，变成大范围变更。

所以演示里 `select * from customer_info where 1=1;` 会被直接阻断。正式项目里也可以按客户制度调整成阻断、审批或告警。

### Q18：规则策略到底怎么生效？

可以用三句话解释：

1. 平台先解析 SQL，得到 SQL 类型、目标库、目标表、字段、WHERE、方言、风险特征。
2. 规则引擎拿这些结构化信息去匹配规则条件。
3. 命中后输出处置结果：放行、提示、脱敏、审批或阻断。

举例：

```text
规则条件：SQL 类型 = UPDATE，目标环境 = 生产
处置结果：进入审批
```

所以用户提交生产 UPDATE 时，平台不是靠人工判断，而是规则引擎自动命中。

### Q19：数据资产与脱敏的数据怎么来？

演示版使用预置 mock 数据，覆盖客户、账务、风控、人事薪酬四类业务域。正式环境通常来自三类来源：

- 数据库元数据扫描：库、Schema、表、字段、类型、行数。
- 敏感字段识别：按字段名、注释、样本规则或数据分类模型识别手机号、证件号、银行卡、设备指纹等。
- 人工治理补充：安全管理员确认敏感等级、脱敏规则、责任团队和启停状态。

一句话口径：

> 系统先自动发现，再由安全或数据负责人确认，最后进入脱敏和审计策略。

### Q20：客户希望有完整 PL/SQL 能力，这期怎么说？

这期建议只承诺轻量入口，不承诺替代完整 PL/SQL Developer。

当前已覆盖：

- 数据源资源树
- 表结构详情
- 受控 SQL 执行
- 查询结果预览
- 执行历史回填

后续可扩展：

- SQL 格式化和收藏
- 执行计划
- 对象详情增强
- 脚本批量执行
- 会话和锁查看
- 结果导出审批

对客户可以这样说：

> 这期先保证生产访问管控闭环，轻量客户端解决常用查询和对象查看。完整 PL/SQL 能力可以作为增强模块继续规划，但安全审核、审批、脱敏、审计不能被绕过。

### Q21：Schema 和表很多时，资源树展示会不会有性能问题？

回答口径：

> 正式系统不会一次性把所有库、所有 Schema、所有表全量加载到浏览器。资源树会先按用户权限裁剪，再按数据源、Schema、表逐层懒加载，用户展开到哪一层才加载哪一层。

可以补充四点：

- 权限裁剪：用户只能看到授权数据源和授权对象，不展示无关库表。
- 分层加载：先加载数据源和 Schema，展开 Schema 时再加载表。
- 缓存与搜索：常用元数据会缓存，表很多时通过搜索、分页或虚拟滚动定位。
- 元数据同步：资源树读的是平台元数据目录，不是每次打开页面都扫生产库。

一句话总结：

> 大规模场景下，资源树展示的是“按权限裁剪后的元数据视图”，不是对生产库做实时全量扫描，所以性能和生产库压力都可控。

## 23. 演示后恢复

演示后如需恢复初始数据，打开浏览器控制台执行：

```js
window.__RKLINK_SQL_SECURITY_DEMO__?.resetAll();
```

该操作会恢复：

- 用户和初始密码
- 用户权限和角色
- 数据源连接和驱动
- SQL 规则策略
- 敏感字段目录
- 审批单
- 执行记录
- SQL 客户端执行历史
- 审计日志

执行后刷新页面即可。
